# Clarification Protocol

## 核心规则

- 阶段一必须使用单维度提问
- 必须按固定顺序推进，不要把多个维度揉成一题
- 不得复用选项编码
- 信息未完整前，不得输出“已记下”
- 信息未完整前，不得输出摘要
- 不得从 Figma 推断 API

## 固定顺序

1. `S*` 范围
2. `T*` 目标端
3. `U*` UI / 设计稿状态
4. `A*` API / 后端依赖状态
5. `V*` 验收方式
6. 摘要确认

## 编码示例

### 范围

- `S1`: 仅订单确认页
- `S2`: 订单确认 + 支付选择
- `S3`: 仅支付收银台
- `S4`: 其他，需文字说明

### 目标端

- `T1`: 仅 PC
- `T2`: 仅 H5 / 移动端
- `T3`: PC + H5 或其他多端

### UI / 设计稿

- `U1`: 无 UI 变更
- `U2`: 有现有 PRD / 设计稿
- `U3`: 有新接口或新 Figma，后续需补链接

### API

- `A1`: 不涉及 API
- `A2`: 涉及现有 API
- `A3`: 涉及新增 API

### 验收方式

- `V1`: 用户给出 1-3 条验收条件
- `V2`: 按 PRD / 设计稿验收
- `V3`: 先记录待补充，后续补齐

## 部分回答处理规则

- 只确认当前轮涉及的字段
- 若用户一次回复多个字段，可分别拆分记录，但不得推断缺失字段
- 必须明确指出“已确认字段”和“仍缺字段”
- 例：
  - 已确认：范围 = `S1`
  - 仍缺：目标端、UI 状态、API 状态、验收方式

## 禁止行为

- 禁止把不同轮次的 `A/B/C` 解释成同一组含义
- 禁止把 `C` 这类裸值按上一轮选项语义解释
- 禁止根据 Figma 节点、页面元素、文案名称自动整理出“接口列表”
- 禁止在用户只给 Figma 链接时写“已确认 7 个接口”
- 禁止在字段未齐时说“已整理如下摘要，请确认”

## 进入摘要确认前的最小条件

至少齐全以下字段后，才能进入摘要确认：

- 范围
- 目标端
- UI / 设计稿状态
- API 状态
- 验收方式
